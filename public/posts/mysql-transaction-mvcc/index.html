MySQL 事务隔离级别与 MVCC 机制深度解析

在高并发场景下，多个事务同时读写数据库，如何保证数据的一致性？MySQL 通过事务隔离级别和 MVCC（多版本并发控制）机制，在性能和一致性之间找到了平衡点。本文深入剖析这套机制的实现原理。

并发事务的三大问题#

在没有任何并发控制的情况下，多个事务同时执行会出现三种问题：

脏读（Dirty Read）

事务 A 读取了事务 B 未提交的数据，如果事务 B 回滚，事务 A 读到的就是脏数据。

-- 事务 A
SELECT balance FROM account WHERE id = 1;  -- 读到 500（脏读）

-- 事务 B
ROLLBACK;  -- 回滚，balance 还是 1000

不可重复读（Non-Repeatable Read）

事务 A 两次读取同一行数据，结果不一致（因为事务 B 在中间修改并提交了数据）。

四种事务隔离级别#

SQL 标准定义了四种隔离级别，用于解决上述问题：

隔离级别 脏读 不可重复读 幻读
READ UNCOMMITTED 可能 可能 可能
READ COMMITTED 不可能 可能 可能
REPEATABLE READ 不可能 不可能 可能
SERIALIZABLE 不可能 不可能 不可能

READ UNCOMMITTED（读未提交）

事务可以读取其他事务未提交的数据。性能最高，但会出现脏读，生产环境几乎不用。

READ COMMITTED（读已提交）

事务只能读取其他事务已提交的数据。Oracle 和 PostgreSQL 的默认隔离级别。

实现方式：每次查询都生成一个新的 ReadView，只能看到已提交的数据。

REPEATABLE READ（可重复读）

事务内多次读取同一行数据，结果一致。MySQL InnoDB 的默认隔离级别。

实现方式：事务开始时生成一个 ReadView，整个事务期间都使用这个 ReadView。

SERIALIZABLE（串行化）

所有事务串行执行，完全避免并发问题。性能最差，只在对一致性要求极高的场景使用。

实现方式：对所有读操作加共享锁，对所有写操作加排他锁。

架构思考： 隔离级别的选择是性能和一致性的权衡。READ COMMITTED 性能高但可能出现不可重复读，REPEATABLE READ 一致性好但可能出现幻读，SERIALIZABLE 完全一致但性能差。没有完美的方案，只有最适合业务的方案。

MVCC：多版本并发控制#

MVCC 是 MySQL 实现 READ COMMITTED 和 REPEATABLE READ 隔离级别的核心机制。它的核心思想是：为每一行数据维护多个版本，不同事务看到不同版本的数据。

隐藏列：事务 ID 和回滚指针

InnoDB 为每一行数据添加了三个隐藏列：

  • DB_TRX_ID：最后修改这行数据的事务 ID
  • DB_ROLL_PTR：回滚指针，指向 undo log 中的上一个版本
  • DB_ROW_ID：行 ID（如果表没有主键，InnoDB 会自动生成）

假设有一行数据：

1

id=1, name='张三', DB_TRX_ID=100, DB_ROLL_PTR=NULL

事务 101 修改了这行数据：

1

UPDATE user SET name = '李四' WHERE id = 1;

修改后，当前行变成：

1

id=1, name='李四', DB_TRX_ID=101, DB_ROLL_PTR=指向 undo log

undo log 中保存了旧版本：

1

id=1, name='张三', DB_TRX_ID=100, DB_ROLL_PTR=NULL

这样，不同事务可以通过回滚指针找到自己需要的版本。

ReadView：可见性判断规则

ReadView 是事务在某个时刻对数据库的快照，包含四个关键字段：

  • m_ids：当前活跃的事务 ID 列表
  • min_trx_id：最小的活跃事务 ID
  • max_trx_id：下一个要分配的事务 ID
  • creator_trx_id：创建这个 ReadView 的事务 ID

boolean isVisible(long trxId, ReadView readView) {
    // 1. 如果是当前事务修改的，可见
    if (trxId == readView.creator_trx_id) {
        return true;
    }
    
    // 2. 如果在 ReadView 创建之前就已提交，可见
    if (trxId < readView.min_trx_id) {
        return true;
    }
    
    // 3. 如果在 ReadView 创建之后才开始，不可见
    if (trxId >= readView.max_trx_id) {
        return false;
    }
    
    // 4. 如果在活跃事务列表中，不可见
    if (readView.m_ids.contains(trxId)) {
        return false;
    }
    
    // 5. 其他情况，可见
    return true;
}

READ COMMITTED vs REPEATABLE READ

两者的区别在于 ReadView 的生成时机：

  • READ COMMITTED：每次查询都生成新的 ReadView
  • REPEATABLE READ：事务开始时生成 ReadView，整个事务期间复用

案例：

-- 事务 A（ID=100）
BEGIN;
SELECT * FROM user WHERE id = 1;  -- 生成 ReadView

-- 事务 B（ID=101）
BEGIN;
UPDATE user SET name = '李四' WHERE id = 1;
COMMIT;

-- 事务 A
SELECT * FROM user WHERE id = 1;  -- 读到什么？

在 READ COMMITTED 下：

  • 第二次查询生成新的 ReadView，此时事务 101 已提交
  • 可以看到事务 101 的修改，读到"李四"

在 REPEATABLE READ 下：

  • 第二次查询复用第一次的 ReadView，此时事务 101 还在活跃列表中
  • 看不到事务 101 的修改，读到"张三"

避坑提示： MVCC 只对 SELECT 查询生效，对 UPDATE、DELETE、INSERT 不生效。这些操作会加锁，遵循"当前读"规则。

幻读问题与间隙锁#

MVCC 解决了不可重复读，但无法完全解决幻读。

为什么 MVCC 无法解决幻读？

MVCC 只能保证读到的数据是一致的，但无法阻止其他事务插入新数据。

案例：

-- 事务 A
BEGIN;
SELECT * FROM account WHERE balance > 500;  -- 返回 3 条

-- 事务 B
BEGIN;
INSERT INTO account (id, balance) VALUES (4, 600);
COMMIT;

-- 事务 A
UPDATE account SET balance = balance + 100 WHERE balance > 500;  -- 更新了 4 条！

虽然第一次 SELECT 只看到 3 条记录，但 UPDATE 是当前读，会看到事务 B 插入的新记录。

间隙锁（Gap Lock）

InnoDB 通过间隙锁解决幻读问题。间隙锁锁定的是索引记录之间的间隙，阻止其他事务在间隙中插入数据。

假设有索引值：10, 20, 30

执行查询：

1

SELECT * FROM account WHERE id > 15 AND id < 25 FOR UPDATE;

InnoDB 会加三把锁：

  • 记录锁：锁定 id=20 的记录
  • 间隙锁：锁定 (10, 20) 和 (20, 30) 的间隙

这样，其他事务无法在 (10, 30) 范围内插入数据，避免了幻读。

Next-Key Lock

Next-Key Lock = 记录锁 + 间隙锁，锁定一个左开右闭的区间。

案例：

1

SELECT * FROM account WHERE id >= 20 FOR UPDATE;

加锁范围：(10, 20] + (20, 30] + (30, +∞)

避坑提示： 间隙锁会降低并发性能，甚至导致死锁。如果业务对幻读不敏感，可以考虑使用 READ COMMITTED 隔离级别，避免间隙锁。

-- 事务 B
INSERT INTO account (id, balance) VALUES (8, 100);  -- 等待事务 A 释放 (5, 10]

两个事务互相等待，形成死锁。

死锁排查

查看最近一次死锁信息：

1

SHOW ENGINE INNODB STATUS;

输出包含死锁的事务 ID、SQL 语句、加锁情况。

死锁预防

  1. 按相同顺序访问数据，避免循环等待
  2. 缩小事务范围，减少锁持有时间
  3. 使用 READ COMMITTED 隔离级别，避免间隙锁
  4. 为查询添加合适的索引，减少锁定范围

总结与思考#

MySQL 的事务隔离机制通过 MVCC 和锁实现了并发控制：

  • MVCC 解决了脏读和不可重复读，性能高
  • 间隙锁解决了幻读，但会降低并发性能
  • 隔离级别的选择要根据业务场景权衡

理解这套机制，才能在高并发场景下设计出高性能、高一致性的数据库方案。
