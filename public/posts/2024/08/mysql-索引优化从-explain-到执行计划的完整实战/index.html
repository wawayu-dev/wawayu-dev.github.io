B+ 树：MySQL 索引的底层实现#

MySQL 的 InnoDB 引擎使用 B+ 树作为索引结构。理解 B+ 树的特性，是优化索引的基础。

B+ 树 vs 二叉搜索树

为什么不用二叉搜索树？因为树的高度会影响查询性能。假设有 100 万条数据：

  • 二叉搜索树：高度约为 20（log₂ 1000000），需要 20 次磁盘 IO
  • B+ 树（每个节点 1000 个键）：高度约为 3（log₁₀₀₀ 1000000），只需 3 次磁盘 IO

B+ 树通过增加节点的扇出度（每个节点存储更多键），大幅降低了树的高度。

B+ 树的关键特性

  1. 所有数据都存储在叶子节点，非叶子节点只存储键值
  2. 叶子节点之间通过指针连接，形成有序链表
  3. 每个节点的大小等于一个磁盘页（InnoDB 默认 16KB）

这些特性带来三个优势：

  • 范围查询高效：通过叶子节点的链表，可以快速扫描连续数据
  • 磁盘 IO 少：树的高度低，查询时访问的磁盘页少
  • 缓存友好：非叶子节点只存储键值，可以在内存中缓存更多节点

架构思考： B+ 树的设计充分考虑了磁盘 IO 的特性。磁盘的随机读写速度远低于顺序读写，B+ 树通过降低树高减少随机 IO，通过叶子节点链表实现顺序扫描。这是"软件设计要适配硬件特性"的经典案例。

EXPLAIN：SQL 性能分析的第一步#

优化 SQL 的第一步是用 EXPLAIN 查看执行计划：

1EXPLAIN SELECT * FROM orders WHERE user_id = 123 AND status = 1;

输出结果包含多个字段，我们重点关注以下几个：

type：访问类型

从好到坏依次是：

  • system：表只有一行数据（系统表）
  • const：通过主键或唯一索引查询，最多返回一行
  • eq_ref：连接查询中，被驱动表通过主键或唯一索引关联
  • ref：通过非唯一索引查询
  • range：索引范围扫描（BETWEEN、>、< 等）
  • index：全索引扫描（扫描整个索引树）
  • ALL：全表扫描（最差）

key：实际使用的索引

如果是 NULL，说明没有使用索引，需要优化。

rows：扫描的行数

这是 MySQL 估算的值，不是精确值。rows 越小，查询越快。

Extra：额外信息

  • Using index：覆盖索引，不需要回表查询
  • Using where：在存储引擎层过滤后，还需要在 Server 层再次过滤
  • Using filesort：需要额外排序，性能较差
  • Using temporary：需要创建临时表，性能很差

实战案例：慢查询优化

假设有一张订单表：

1CREATE TABLE orders (
2    id BIGINT PRIMARY KEY,
3    user_id BIGINT,
4    status TINYINT,
5    create_time DATETIME,
6    INDEX idx_user_id (user_id)
7);

执行查询：

1SELECT * FROM orders WHERE user_id = 123 AND status = 1 ORDER BY create_time DESC LIMIT 10;

EXPLAIN 结果：

1type: ref
2key: idx_user_id
3rows: 5000
4Extra: Using where; Using filesort

问题分析：

  1. 使用了 idx_user_id 索引，type 是 ref，还不错
  2. 但 Extra 显示 Using filesort，说明需要额外排序，性能较差
  3. rows 是 5000，说明扫描了 5000 行数据

优化方案：创建联合索引 (user_id, status, create_time)

1ALTER TABLE orders ADD INDEX idx_user_status_time (user_id, status, create_time);

再次 EXPLAIN：

1type: ref
2key: idx_user_status_time
3rows: 10
4Extra: Using index condition

优化效果：

  • rows 从 5000 降低到 10
  • 去掉了 Using filesort，不需要额外排序

联合索引：最左前缀匹配原则#

联合索引的顺序很重要，必须遵循最左前缀匹配原则。

原理解析

假设有联合索引 (a, b, c)，索引的排序规则是：

  1. 先按 a 排序
  2. a 相同时，按 b 排序
  3. a 和 b 都相同时，按 c 排序

这意味着：

  • WHERE a = 1：可以使用索引
  • WHERE a = 1 AND b = 2：可以使用索引
  • WHERE a = 1 AND b = 2 AND c = 3：可以使用索引
  • WHERE b = 2：不能使用索引（缺少 a）
  • WHERE a = 1 AND c = 3：只能使用 a 部分的索引（缺少 b）

索引顺序的选择策略

  1. 区分度高的字段放在前面
  2. 经常用于查询条件的字段放在前面
  3. 经常用于排序的字段放在后面

实战案例：订单查询优化

业务场景：

  • 查询某个用户的订单：WHERE user_id = ?
  • 查询某个用户的待支付订单：WHERE user_id = ? AND status = 1
  • 查询某个用户的订单，按时间倒序：WHERE user_id = ? ORDER BY create_time DESC

索引设计：(user_id, status, create_time)

为什么不是 (user_id, create_time, status)？

因为第二个查询需要同时过滤 user_id 和 status，如果 status 在最后，就无法利用索引的有序性。

避坑提示： 联合索引不是越多越好。每增加一个索引，都会增加写入时的维护成本（插入、更新、删除都需要更新索引）。建议一张表的索引数量控制在 5 个以内。

覆盖索引：避免回表查询#

什么是回表？

InnoDB 的索引分为两种：

  • 聚簇索引（主键索引）：叶子节点存储完整的行数据
  • 二级索引（普通索引）：叶子节点只存储索引列和主键值

当通过二级索引查询时，如果需要的列不在索引中，就需要回表：

  1. 先通过二级索引找到主键值
  2. 再通过主键值到聚簇索引中查找完整行数据

回表需要额外的磁盘 IO，影响性能。

覆盖索引优化

如果查询的列都在索引中，就不需要回表，这叫覆盖索引。

案例：

1-- 需要回表
2SELECT * FROM orders WHERE user_id = 123;
3
4-- 不需要回表（覆盖索引）
5SELECT id, user_id, status FROM orders WHERE user_id = 123;

如果有索引 (user_id, status)，第二个查询就是覆盖索引（查询列 id、user_id、status 都在索引中）。

EXPLAIN 结果会显示 Extra: Using index。

实战技巧：索引列冗余

有时候为了实现覆盖索引，可以在索引中冗余一些常用列：

1-- 原索引
2ALTER TABLE orders ADD INDEX idx_user_id (user_id);
3
4-- 优化后：冗余 status 和 create_time
5ALTER TABLE orders ADD INDEX idx_user_id_status_time (user_id, status, create_time);

这样，大部分查询都能命中覆盖索引，避免回表。

索引下推：减少回表次数#

MySQL 5.6 引入了索引下推（Index Condition Pushdown，ICP）优化。

没有索引下推的情况

假设有索引 (user_id, status)，执行查询：

1SELECT * FROM orders WHERE user_id = 123 AND status > 0;

传统流程：

  1. 通过索引找到所有 user_id = 123 的记录
  2. 回表获取完整行数据
  3. 在 Server 层过滤 status > 0

问题：即使 status 不满足条件，也会回表，浪费 IO。

有索引下推的情况

优化后的流程：

  1. 通过索引找到所有 user_id = 123 的记录
  2. 在索引层就过滤 status > 0（不需要回表）
  3. 只对满足条件的记录回表

EXPLAIN 结果会显示 Extra: Using index condition。

索引下推的好处：减少回表次数，降低磁盘 IO。

架构思考： 索引下推体现了"计算下推"的思想：把过滤逻辑尽可能下推到存储引擎层，减少数据传输量。类似的思想还有：数据库的谓词下推、大数据的算子下推。

索引失效的常见场景#

即使创建了索引,某些情况下也会失效：

1. 在索引列上使用函数

1-- 索引失效
2SELECT * FROM orders WHERE DATE(create_time) = '2024-08-10';
3
4-- 优化：改写为范围查询
5SELECT * FROM orders WHERE create_time >= '2024-08-10 00:00:00' 
6  AND create_time < '2024-08-11 00:00:00';

2. 隐式类型转换

1-- user_id 是 BIGINT 类型，传入字符串会导致索引失效
2SELECT * FROM orders WHERE user_id = '123';
3
4-- 优化：传入正确类型
5SELECT * FROM orders WHERE user_id = 123;

3. 使用 OR 连接不同索引列

1-- 索引失效（user_id 和 status 在不同索引中）
2SELECT * FROM orders WHERE user_id = 123 OR status = 1;
3
4-- 优化：改写为 UNION
5SELECT * FROM orders WHERE user_id = 123
6UNION
7SELECT * FROM orders WHERE status = 1;

4. 模糊查询以 % 开头

1-- 索引失效
2SELECT * FROM users WHERE name LIKE '%张三%';
3
4-- 部分优化：如果只是后缀模糊，可以使用索引
5SELECT * FROM users WHERE name LIKE '张三%';

总结与思考#

MySQL 索引优化的核心是：理解 B+ 树的存储结构，利用索引的有序性，减少磁盘 IO。

  • 通过 EXPLAIN 分析执行计划，找到性能瓶颈
  • 设计合理的联合索引，遵循最左前缀匹配原则
  • 利用覆盖索引避免回表，利用索引下推减少回表次数
  • 避免索引失效的常见陷阱

索引不是越多越好，每个索引都有维护成本。在设计索引时，要结合业务场景，权衡查询性能和写入性能。

下次当你的 SQL 查询很慢时，不妨先 EXPLAIN 一下，看看是不是索引的问题。
