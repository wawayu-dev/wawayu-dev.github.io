# 写作指南

> 确保所有文章保持一致的质量和风格

---

## 🎯 写作目标

### 文章定位
- **目标读者**：中高级 Java 开发者
- **内容深度**：源码分析 + 实战案例
- **写作风格**：技术专业 + 通俗易懂

### 核心价值
1. **深度**：不止于使用，深入原理
2. **实战**：结合真实场景，提供可落地的方案
3. **思考**：分析设计思想，提升架构能力

---

## 📝 写作规范

### 标题规范

**文章标题格式**
```
技术主题 + 核心内容 + 深度标识

示例：
✅ "Spring AOP 原理与动态代理实现深度解析"
✅ "MySQL 索引优化：从 Explain 到执行计划"
✅ "Redis 分布式锁的生产级实践：从原理到 Redisson"

❌ "Spring AOP 教程"（太笼统）
❌ "如何使用 Redis"（太基础）
```

**章节标题规范**
- 二级标题：核心概念、实战应用、最佳实践
- 三级标题：具体技术点、场景描述
- 避免使用"前言"、"介绍"等无意义标题

---

### 开场白规范

**结构**
```markdown
第1段：背景介绍（为什么需要这个技术）
第2段：问题描述（要解决什么问题）
第3段：文章内容（本文会讲什么）

### 本文亮点
- [x] 亮点1
- [x] 亮点2
- [x] 亮点3
```

**示例**
```markdown
在分布式系统中，多个服务实例可能同时访问共享资源，需要分布式锁来保证数据一致性。Redis 是实现分布式锁的常用方案，但很多人只会用 SETNX，不知道其中的坑。本文深入剖析 Redis 分布式锁的实现。

### 本文亮点
- [x] 理解 Redis 分布式锁的实现原理与常见问题
- [x] 掌握 Redisson 看门狗机制的设计思想
- [x] 学会高并发场景下的锁优化方案
- [x] 了解分布式锁的最佳实践
```

---

### 正文规范

**内容组织**
1. **从简单到复杂**：先讲基础概念，再讲高级特性
2. **理论+实践**：先讲原理，再给代码示例
3. **问题导向**：围绕实际问题展开，不要堆砌知识点

**段落规范**
- 每段3-5句话
- 一段只讲一个观点
- 段落之间有逻辑关系

**列表规范**
- 使用列表突出要点
- 列表项保持格式一致
- 避免过长的列表（超过10项）

---

### 代码示例规范

**代码质量要求**
1. **可运行**：代码必须能直接运行
2. **完整性**：包含必要的导入和类定义
3. **简洁性**：只保留核心逻辑
4. **注释性**：关键代码添加注释

**代码格式**
\```java
// 1. 添加场景说明注释
// 2. 突出关键代码

public class Example {
    /**
     * 方法说明：做什么，为什么这样做
     */
    public void method() {
        // 步骤1：xxx
        step1();
        
        // 步骤2：xxx
        step2();
    }
}
\```

**对比示例**
\```java
// 不好的做法
public void badExample() {
    // 问题说明
}

// 好的做法
public void goodExample() {
    // 改进说明
}
\```

### 特殊格式规范

**架构思考**
```markdown
> **架构思考：** 分析设计思想、权衡、原理等深度内容。

示例：
> **架构思考：** MVCC 的设计体现了"空间换时间"的思想。通过 Undo Log 保存历史版本，牺牲了存储空间，换取了读写并发的性能提升。
```

**避坑提示**
```markdown
> **避坑提示：** 常见问题、注意事项、最佳实践。

示例：
> **避坑提示：** 在使用线程池的环境下，ThreadLocal 必须在 finally 块中调用 remove()。否则会导致内存泄漏和数据混乱。
```

### 总结规范

**结构**
```markdown
## 总结与思考

核心要点总结（3-5个）：
- 要点1
- 要点2
- 要点3

思考与展望：
下次当你遇到xxx问题时，不妨试试本文介绍的方法。
```

**要求**
- 总结要简洁，不要重复正文
- 提炼核心观点，不要流水账
- 给读者留下思考空间

## 🎨 写作风格

### 语言风格

**专业但不生硬**
```
✅ "Spring 通过 BeanPostProcessor 实现了扩展点"
❌ "Spring 用了 BeanPostProcessor 这个东西"
```

**通俗但不口语化**
```
✅ "这个设计很巧妙"
❌ "这个设计贼牛逼"
```

**简洁但不省略**
```
✅ "通过 CAS 操作保证原子性"
❌ "用 CAS"（太简略）
❌ "通过 Compare And Swap 操作来保证操作的原子性"（太啰嗦）
```

### 术语使用

**中英文混用规范**
- 首次出现：中文（英文缩写）
- 后续使用：英文缩写或中文

```
示例：
首次：多版本并发控制（MVCC）
后续：MVCC、多版本并发控制
```

**专业术语**
- 使用行业标准术语
- 避免自创术语
- 必要时给出解释

## 🚫 常见问题

### 避免的写作方式

❌ **堆砌知识点**
```
不要：列举所有 API，没有重点
要做：围绕核心问题，深入讲解
```

❌ **流水账式**
```
不要：第一步xxx，第二步xxx，第三步xxx
要做：说明为什么这样做，背后的原理是什么
```

❌ **过于简单**
```
不要：只讲怎么用，不讲为什么
要做：深入原理，分析设计思想
```

❌ **过于复杂**
```
不要：大段理论，没有实例
要做：理论+实践，通俗易懂
```
